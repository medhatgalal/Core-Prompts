# 02 Research — 2026 Uplift Engine (Context/Intent/Task Decomposition/Constraints/Acceptance)

Date: 2026-03-04
Phase: 02-2026-uplift-engine-context-intent-task-decomposition-constraints-acceptance
Scope lock: Context Layer, Intent Layer, Task Decomposition, Constraint Architecture, Acceptance Criteria.

## Research Outcome

Phase 2 should introduce a deterministic, typed uplift engine that consumes sanitized text and emits one canonical contract object:
- `context`
- `intent`
- `task_graph`
- `constraints`
- `acceptance`

The engine should be pure and side-effect free. No network calls, tool execution, Rosetta translation, output generation, or target-tool validation should exist in this phase path.

## Existing Baseline To Preserve

From current `src/intent_pipeline` and tests:
- Use pure functions with explicit `str`/typed input validation and deterministic return values.
- Use typed errors with explicit error codes (Enum-backed) and structured details.
- Preserve stable ordering and deterministic serialization behavior.
- Keep phase boundaries explicit in public API shape (as done in Phase 1 boundary tests).

## Proposed Contract and Schema Strategy

Use a versioned schema contract with strict major version gating.

Recommended top-level model (implementation can be dataclasses now, JSON schema export later):

```python
@dataclass(frozen=True)
class UpliftContract:
    schema_version: str
    context: ContextModel
    intent: IntentModel
    task_graph: TaskGraph
    constraints: ConstraintSet
    acceptance: AcceptanceReport
```

Rules:
- `schema_version` is required and validated by major version (`2.x`).
- All collections are materialized in deterministic order before return.
- Inferred fields must carry provenance markers (`source="inferred"`, `confidence`, `evidence_ids`).
- Use integers for scoring where possible (`0..100`) to avoid float drift.

## Architecture and Module Plan

Create a new package under `src/intent_pipeline/uplift/` with narrowly scoped modules:

- `contracts.py`
  - Dataclasses/enums for all phase-2 structures.
  - Error classes and deterministic rejection/violation codes.
- `context_layer.py`
  - Build layered context object from sanitized text and deterministic extraction rules.
- `intent_layer.py`
  - Convert context into normalized explicit intent representation.
- `task_decomposition.py`
  - Build a two-level DAG with stable node IDs and canonical topological serialization.
- `constraints.py`
  - Parse/classify constraints (`hard`/`soft`), detect conflicts, and resolve deterministically.
- `acceptance.py`
  - Evaluate criteria with pass/fail/needs-review and criterion-linked evidence.
- `engine.py`
  - Compose stages and enforce phase-2 boundary contract.

Keep `src/intent_pipeline/pipeline.py` phase-1 behavior unchanged. Add a new phase-2 entrypoint instead of expanding the phase-1 terminal function.

## Layer Implementation Guidance

### 1) Context Layer

Goal: Produce a layered context object with no silent merging of inferred data.

Recommended shape:
- `source`: canonical input metadata (sanitized input hash, line count, schema version).
- `normalized`: normalized facts extracted with deterministic rules.
- `inferred`: inferred facts with provenance and confidence.
- `constraints`: extracted candidate constraints before conflict resolution.
- `acceptance_inputs`: extracted candidate acceptance criteria/evidence anchors.

Deterministic extraction rules:
- Normalize line tokens once, then classify by fixed precedence rule list.
- Use stable line index as tiebreaker when multiple matches occur.
- Never infer when explicit evidence exists for the same canonical key.

### 2) Intent Layer

Goal: Convert context into stable intent primitives planner can consume directly.

Recommended model:
- `primary_objective: str`
- `secondary_objectives: list[str]`
- `in_scope: list[str]`
- `out_of_scope: list[str]`
- `quality_constraints: list[str]`
- `unknowns: list[str]`

Rules:
- Populate `unknowns` when evidence is missing instead of guessing.
- Canonicalize text deterministically (whitespace/case policy fixed in one helper).
- Deduplicate with insertion-preserving uniqueness.

### 3) Task Decomposition

Goal: Produce an atomic, testable, dependency-aware DAG with depth <= 2.

Deterministic graph rules:
- Node depth only `1` (task) or `2` (subtask).
- Node ID generated by deterministic sequence after canonical sort (`T001`, `T002`, `T002.1`).
- Edges are explicit dependency IDs; no implicit ordering assumptions.
- Serialization order: topological order, then `(priority_rank, lexical_title, original_index)` tiebreak.

Conflict-safe behavior:
- Cycles are hard violations with typed failure (`TASK_GRAPH_CYCLE`).
- Overspecified depth (>2) is rejected with structured validation error.

### 4) Constraint Architecture

Goal: Represent constraints in typed structured fields and resolve conflicts deterministically.

Constraint type:
- `constraint_id`
- `severity` (`HARD` | `SOFT`)
- `subject`
- `operator`
- `value`
- `source` (`explicit` | `inferred`)
- `evidence_ids`

Precedence policy:
1. Hard explicit
2. Hard inferred
3. Soft explicit
4. Soft inferred

Resolution policy:
- Hard-vs-hard contradiction: fail with structured conflict payload.
- Hard-vs-soft contradiction: hard wins, soft marked overridden.
- Soft-vs-soft contradiction: choose deterministic winner by `(precedence, source_line, lexical constraint_id)` and report both.

### 5) Acceptance Criteria

Goal: Evaluate criterion-level readiness with fixed thresholds and explicit evidence links.

Acceptance model:
- `decision`: `PASS` | `FAIL` | `NEEDS_REVIEW`
- `score`: integer `0..100`
- `criteria`: list of criterion evaluations including `criterion_id`, `status`, `evidence_ids`, `task_ids`, `rationale`
- `gates`: hard-gate results and reasons

Deterministic decision logic:
- Any unmet hard criterion => `FAIL`.
- Missing required evidence for any hard criterion => `NEEDS_REVIEW`.
- Otherwise compute integer score via fixed formula and compare to fixed threshold.
- Use strict comparator and fixed threshold constant in code (`ACCEPTANCE_PASS_THRESHOLD = 80`).

## Validation Architecture

Build requirement-indexed tests in `tests/` using existing deterministic style.

### Requirement IDs for Phase 2

- `CTX-01`: context model includes required layered sections.
- `CTX-02`: inferred values are explicitly marked and never overwrite explicit facts.
- `INT-01`: intent model is deterministically derived from context.
- `INT-02`: unknown/ambiguous fields map to `unknowns` rather than guessed values.
- `TASK-01`: decomposition generates depth-limited DAG (`<=2`).
- `TASK-02`: DAG serialization order is deterministic and stable across runs/processes.
- `CON-01`: constraints are typed (`hard`/`soft`) with structured fields.
- `CON-02`: conflicts resolve by deterministic precedence with explicit conflict reports.
- `ACC-01`: acceptance emits criterion-level evidence linked to task IDs.
- `ACC-02`: acceptance decision follows fixed gate+threshold logic.
- `ACC-03`: ambiguous/incomplete evaluation returns `NEEDS_REVIEW`.
- `BOUND-03`: phase-2 engine excludes Rosetta translation, target tool validation, output generation.

### Test Layout Recommendation

- `tests/test_phase2_context_layer.py`
- `tests/test_phase2_intent_layer.py`
- `tests/test_phase2_task_decomposition.py`
- `tests/test_phase2_constraints.py`
- `tests/test_phase2_acceptance.py`
- `tests/test_phase2_boundary.py`
- `tests/test_phase2_determinism.py`

### Determinism Harness

For each fixture:
- Run the full phase-2 engine in-process multiple times and assert byte-identical serialized JSON.
- Run the engine in fresh Python processes (with `PYTHONHASHSEED=0`, fixed locale/TZ) and assert byte-identical output.
- Hash outputs with SHA-256 and assert single unique digest.

### Validation Failure Shape

When hard validation fails, return deterministic structured payloads:
- `code` (Enum value)
- `message` (stable wording)
- `details` (ordered key-value fields)

This mirrors current policy/error patterns and keeps failures planner-actionable.
