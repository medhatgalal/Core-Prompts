# Content Intake Template

Use this template before any landing decision.

## <Content Title> ##

### Intent
- `new_ssot_capability` | `uplift_existing_ssot` | `research_only`

### Purpose
- What problem this family solves.
- When it should be invoked.

### Goals
- Concrete outcomes the capability or research should achieve.

### Sources
- Primary external sources
- Local benchmark sources

### Research Targets
- Additional sources worth inspecting before landing

### Benchmarks In Repo
- Existing SSOT entries or docs that define the current quality bar

### Current Gaps
- What is missing today in the repo
- Why current capabilities are insufficient

### Proposed Public Name
- Human-facing name

### Proposed Slug
- Canonical slug if landed

### Proposed Description
- Short description optimized for intent routing and capability selection

### Capability Type
- `skill` | `agent` | `both` | `manual_review`

### Role / Invocation Hints
- The kinds of user asks that should trigger this family

### Required Inputs
- Information the capability needs to produce good output

### Expected Outputs
- Deliverables and artifact shapes

### Tool and Runtime Boundaries
- Allowed actions
- Forbidden actions
- Explicit non-goals

### Metadata Expectations
- Display name, summary fidelity, advisory-only boundary, expected descriptor fields

### Surface Expectations
- Which surfaces should emit if landed

### Acceptance Criteria
- What must be true before the family is considered ready

### Evaluation Examples
- One good example request
- One good example output shape
- One failure mode to guard against

### Recommended Landing Paths
- SSOT path
- descriptor path
- docs or planning paths if not landing as SSOT

### Open Risks
- Drift risks
- source-quality risks
- overlap risks
